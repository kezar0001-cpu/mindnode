# Data Model

All identifiers are UUIDs generated by `gen_random_uuid()` (`pgcrypto`). Timestamps are `timestamptz` and default to `now()`.

Canonical sources:

- [`supabase/migrations/20260528000000_initial_schema.sql`](../supabase/migrations/20260528000000_initial_schema.sql) — initial tables, indexes, trigger.
- [`supabase/migrations/20260528010000_add_auth_and_rls.sql`](../supabase/migrations/20260528010000_add_auth_and_rls.sql) — adds `user_id`, enables RLS, defines per-user CRUD policies.

## User Ownership

Every row in the five core tables belongs to exactly one user. Each table has a `user_id uuid NOT NULL` column referencing `auth.users(id)` with `ON DELETE CASCADE`. Deleting a user (via the Supabase dashboard or `auth.admin.deleteUser()`) wipes that user's MindNode data.

The application is single-user in spirit, but multi-user at the database layer so that authentication and RLS protect personal data when the project is deployed to a hosted Supabase instance.

## Row Level Security

RLS is enabled on all five core tables. Each table has four policies, all keyed on `auth.uid() = user_id`:

| Action | Policy clause |
| ------ | ------------- |
| select | `USING (auth.uid() = user_id)` |
| insert | `WITH CHECK (auth.uid() = user_id)` |
| update | `USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id)` |
| delete | `USING (auth.uid() = user_id)` |

Practical consequences:

- The `authenticated` role can only read/write rows it owns.
- The `anon` role sees no MindNode rows at all.
- The `service_role` key (used only by `src/lib/supabase/admin.ts`) bypasses RLS — keep it server-side.
- New inserts must include `user_id`. The application code in `src/lib/supabase/server.ts` runs as the signed-in user, so application inserts should set `user_id = current user's id`.

## Core Entities

### `memory_entries`

The raw, unprocessed thought as the user typed it. Append-only by convention. The `node_memory_links` foreign key uses `ON DELETE RESTRICT`, so once a memory entry is linked to a node the database refuses to delete it.

| Column      | Type        | Notes                                                     |
| ----------- | ----------- | --------------------------------------------------------- |
| id          | uuid (PK)   | `gen_random_uuid()`                                        |
| user_id     | uuid (FK)   | references `auth.users(id)`, `ON DELETE CASCADE`           |
| content     | text        | the raw input, exactly as typed                            |
| source      | text        | `chat` for MVP; later: `voice`, `import`                   |
| metadata    | jsonb       | open-ended bag for client/source-specific fields           |
| created_at  | timestamptz |                                                            |

### `nodes`

A processed concept on the canvas. Always derived from one or more memory entries via `node_memory_links`.

| Column      | Type             | Notes                                       |
| ----------- | ---------------- | ------------------------------------------- |
| id          | uuid (PK)        |                                             |
| user_id     | uuid (FK)        | references `auth.users(id)`, `ON DELETE CASCADE` |
| title       | text             | short label shown on the canvas             |
| summary     | text             | AI-written summary; never replaces raw text |
| category    | text             | e.g. `idea`, `goal`, `reflection`           |
| position_x  | double precision | canvas coordinate                           |
| position_y  | double precision | canvas coordinate                           |
| created_at  | timestamptz      |                                             |
| updated_at  | timestamptz      | refreshed by `nodes_set_updated_at` trigger |

### `edges`

A directed relationship between two nodes.

| Column            | Type             | Notes                                   |
| ----------------- | ---------------- | --------------------------------------- |
| id                | uuid (PK)        |                                         |
| user_id           | uuid (FK)        | references `auth.users(id)`, `ON DELETE CASCADE` |
| source_node_id    | uuid (FK)        | references `nodes.id`, `ON DELETE CASCADE` |
| target_node_id    | uuid (FK)        | references `nodes.id`, `ON DELETE CASCADE` |
| relationship_type | text             | e.g. `related`, `supports`, `blocks`    |
| label             | text (nullable)  | optional human-readable label            |
| strength          | double precision | weighting, default `1`                  |
| created_at        | timestamptz      |                                         |

Constraint: `source_node_id <> target_node_id` (no self-loops).

### `ai_suggestions`

One row per AI proposal. Stored regardless of whether it's accepted, so we keep a record of how the graph evolved. The full structured payload (action, title, summary, category, confidence, related node IDs, suggested edges, explanation) lives inside `suggestion_json`.

| Column          | Type        | Notes                                                                |
| --------------- | ----------- | -------------------------------------------------------------------- |
| id              | uuid (PK)   |                                                                      |
| user_id         | uuid (FK)   | references `auth.users(id)`, `ON DELETE CASCADE`                     |
| memory_entry_id | uuid (FK)   | references `memory_entries.id`, `ON DELETE CASCADE`                  |
| suggestion_json | jsonb       | full validated suggestion payload (see `docs/AI_BEHAVIOUR.md`)        |
| status          | text        | `pending` / `accepted` / `rejected` (check constraint)               |
| created_at      | timestamptz |                                                                      |
| accepted_at     | timestamptz | nullable; set when status flips to `accepted`                        |

### `node_memory_links`

Many-to-many between nodes and the memory entries that informed them. A node can have multiple linked memory entries (e.g. when an update merges in a new thought). Composite uniqueness on `(node_id, memory_entry_id)` prevents duplicate links.

| Column          | Type        | Notes                                                       |
| --------------- | ----------- | ----------------------------------------------------------- |
| id              | uuid (PK)   |                                                             |
| user_id         | uuid (FK)   | references `auth.users(id)`, `ON DELETE CASCADE`            |
| node_id         | uuid (FK)   | references `nodes.id`, `ON DELETE CASCADE`                  |
| memory_entry_id | uuid (FK)   | references `memory_entries.id`, `ON DELETE RESTRICT`         |
| created_at      | timestamptz |                                                             |

- `memory_entries (user_id)`
- `nodes (category)`
- `nodes (user_id)`
- `edges (source_node_id)`
- `edges (target_node_id)`

- `ai_suggestions (memory_entry_id)`

- `node_memory_links (node_id)`
- `node_memory_links (memory_entry_id)`
- `node_memory_links (user_id)`

## Triggers

- `nodes_set_updated_at` — refreshes `nodes.updated_at` on every `UPDATE`.

## Invariants

- Every `nodes` row should have at least one `node_memory_links` row (enforced at the application layer when accepting a suggestion).
- `memory_entries` rows are never deleted by application code. The `ON DELETE RESTRICT` on `node_memory_links.memory_entry_id` adds a database-level safety net the moment a node references the entry.
- Accepting an AI suggestion should write to `nodes` / `edges` / `node_memory_links` and flip the suggestion's `status` and `accepted_at` in the same transaction.
- All five tables' `user_id` columns must equal `auth.uid()` for both selects and writes — enforced by RLS.

## Out of Scope (for now)

- Teams, organisations, roles, shared canvases — not in MVP and not planned.
- `node_embeddings` (pgvector) for semantic related-node lookup.
- Tags or richer categorisation.
- Soft archive flag on nodes; raw entries still never deleted.
